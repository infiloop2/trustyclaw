# Outbound request parameter guard

`host/param_guard.py` implements the guard this document describes; the two
change together, and this document is the reference for what is checked and
why.

## Overview

One host-owned check applied to every agent-controlled free-text parameter of
an automatic action bound for a public or third-party destination:

```python
api.outbound.guard_request_parameter_string(value)                        # public / third-party
api.outbound.guard_request_parameter_string(value, allow_identifiers=True) # connected-account query
```

The default form is for values sent to a public or third-party destination.
`allow_identifiers=True` is the one sanctioned call-site variation, for a
query against an account the operator already connected (a Gmail mailbox
search): it skips only the personal-identifier rules, because an email
address or phone number is legitimate search syntax against an account that
already holds that data (`from:alice@example.com` must not be denied), while
secret/credential shapes and encoded payloads are still denied. Apart from
that flag the caller passes the value and nothing else — no limits, thresholds, guard
selections, or policy objects — so a call site cannot weaken the check;
tightening or loosening is a host-side reviewed change. The proxy's
network-integration guards run the same rules over decoded request-URL
values (`request_param_denial` in `host/network_integrations/base.py`).

Any finding denies the action. The guard never redacts, truncates, or
rewrites: a silently modified query or side effect is an action the caller
did not request. Every denial returns a **descriptive, value-free error** the
agent can act on — the tool error names what was detected ("appears to
contain an email address … remove it and retry"), the proxy 403 body carries
the reason code, and the network-introspection tools join that code to its
guidance. The value, matched span, and offsets never appear in errors, logs,
or metrics.

All rules are deterministic and offline: expressions compile at import, word
data is a generated checked-in module
(`host/param_guard_words.py`, regenerated by
`tests/generate_param_guard_words.py` from pinned packages), and nothing is
fetched at runtime.

## Guards

Every guard, with how it is implemented. Ids are a documentation grouping —
generic guards first, keyword-adjacency guards together at the end — not an
execution order; the implementation orders checks for cheapness and for
message quality (specific rules before generic ones) and reports the first
finding.

### Generic guards (G1–G17)

Examples are drawn from the tested vectors in `tests/test_param_guard.py`:
✗ denied, ✓ allowed, one example per line.

| ID | Guard | Implementation | Examples | Skipped by `allow_identifiers` |
| --- | --- | --- | --- | --- |
| G1 | `LENGTH` | UTF-8 byte length > **1,024** (fixed host constant) denies. Always applied; the floor that works when every other guard misses — encoding pays a size tax, never a discount. | ✗ a 3 KB email thread pasted as a query<br>✓ any value ≤ 1,024 bytes | no |
| G2 | `PRINTABLE` | Any character in a Unicode `C*` ("Other") category denies, except tab/newline/CR. `C*` covers `Cc` control characters (NUL, ESC, backspace), `Cf` invisible formatting characters (zero-width space U+200B, zero-width joiner, right-to-left override U+202E), `Co` private-use, `Cs` surrogates, and `Cn` unassigned codepoints. | G9 | `CRYPTO_KEY` | 64-hex runs (exact Ethereum/ed25519 private-key length) and WIF-shaped base58 (`5/K/L` + 50–51 base58 chars). | G10 | `CRED_URL` | URL-shaped tokens are parsed: userinfo (`user:pass@host`) denies; a value of 16+ chars under a credential-named query key denies. | G11 | `DIGIT_RUN` | Unbroken runs of **10–16** digits deny — the window where bare phones, cards, and account numbers live. Shorter runs (one-time-code length, dates, order numbers) and longer runs (17+ — the 64-bit provider-id range: tweet, Discord, Meta ids) are allowed; separator-formatted financial numbers are caught by the checksummed rules, valid 17–19-digit cards still fail Luhn+issuer first, and keyword-adjacent short codes are G20's job. | ✗ `call 4155552671` (10 bare digits)<br>✗ `account 12345678901234`<br>✗ `1111111111111117` (16 digits; passes Luhn but no issuer prefix, so the card rule leaves it to this one)<br>✓ `id 482913 lookup` (code-length runs allowed bare)<br>✓ `order 48291365 status`<br>✓ `1846923871234567890` (a pasted tweet id — 17+ allowed)<br>✓ `olympics 2028` / `zip 90210`<br>✓ digit runs inside an exempt public wallet address | yes |
| G12 | `CARD` | Candidates of 13–19 digits with optional single space/dash separators, normalized, must pass **Luhn + issuer prefix + length** (Visa/MC/Amex/Discover/JCB/Diners/UnionPay) to deny. Luhn alone accepts ~1 in 10 digit runs, so the issuer check and a provider-identifier negative corpus in the tests are load-bearing. | ✗ `4111 1111 1111 1111`<br>✗ `5500-0000-0000-0004`<br>✓ `visa vs mastercard fees`<br>✓ `1111111111111117` passes Luhn but has no issuer prefix, so G11 skips it (G10 denies the bare run instead) | yes |
| G13 | `SSN` | Formatted `ddd-dd-dddd` after invalid-range filtering (no 000/666/9xx area, 00 group, 0000 serial). Bare nine-digit forms fall to G10/G13. | ✗ `ssn 219-09-9999`<br>✓ `000-12-3456` (invalid area; each group under the bare-run floor) | yes |
| G14 | `ABA` | Nine-digit runs passing the ABA checksum (3-7-1 weighting) with a valid Federal Reserve district prefix. | ✗ `routing 021000021 wire`<br>✓ `aba routing number lookup` | yes |
| G15 | `IBAN` | `CC` + 2 check digits + grouped body; country-specific length table then ISO 7064 mod-97 == 1. | ✗ `DE89 3704 0044 0532 0130 00`<br>✓ `DE89 3704 account` (too short for any IBAN country) | yes |
| G16 | `EMAIL` | Address-shaped pattern (local part, domain labels, alphabetic TLD). | ✗ `find linkedin of alice.smith@acme-corp.co.uk`<br>✓ `@acme on twitter` (a handle, not an address)<br>✓ `email deliverability tips` | yes |
| G17 | `PHONE` | International `+…` candidates and separator-formatted candidates (`(415) 555-2671`) with 8–15 digits. Bare digit runs fall to G10. | ✗ `call +1 415 555 2671`<br>✗ `(415) 555-2671`<br>✓ `call 911`<br>✓ `flight UA415 status` | yes |

### Keyword-adjacency guards (G18–G22)

These fire only when the pattern sits within a short window of trigger
keywords, which is what lets them use looser shapes safely.

| ID | Guard | Implementation | Examples | Skipped by `allow_identifiers` |
| --- | --- | --- | --- | --- |
| G18 | `ENTROPY_NEAR_KEYWORD` | A token of 16+ base64ish chars with Shannon entropy ≥ 3.2 bits/char within 40 chars of one of these keywords (case-insensitive, whole words; spaces, underscores, or hyphens accepted inside the compounds): `api key` / `api_key` / `apikey`, `access key` / `access_key` / `accesskey`, `private key` / `private_key` / `privatekey`, `secret`, `token`, `bearer`, `credential`. Entropy alone is never a denial (hashes and cursors are normal traffic); the keyword supplies the precision. | ✗ `api key x7Kp2mQv9zR4tYw8LbN3`<br>✓ `how to get an api key for brave search` (keyword, but no high-entropy token)<br>✓ the same random token with no keyword nearby is judged by G18 instead | no |
| G19 | `PASSWORD_KEYWORD` | One of the keywords `password`, `passwd`, `pwd` (case-insensitive, whole words) followed by a connective (`is`, `:`, `=`) and then any value of 4+ non-space chars. The connective keeps "how to change password on iphone" a valid query. | ✗ `password: hunter2secret`<br>✗ `my password is hunter2secret`<br>✓ `how to change password on iphone`<br>✓ `reset password help` | no |
| G20 | `OTP_KEYWORD` | A 4–8 digit run within 24 chars after one of these keywords (case-insensitive, whole words): `code`, `otp`, `verification`, `verify`, `2fa`, `passcode`, `pin`, `one-time` / `one time`. `code` is suppressed when preceded by a location/status qualifier: `zip code`, `area code`, `postal code`, `country code`, `dial code`, `error code`, `status code`, `http code`. With the keyword present the threshold safely reaches below the bare digit window ("PIN 1234"). | ✗ `verification code 482913`<br>✗ `PIN 1234`<br>✓ `zip code 90210 homes`<br>✓ `error code 1001 windows`<br>✓ `area code 415 businesses` | yes |
| G21 | `DOB_KEYWORD` | A date shape (`d/m/y` with `/`, `.`, or `-` separators, or ISO `yyyy-mm-dd`) within 24 chars after one of these keywords (case-insensitive): `dob`, `date of birth`, `born`, `birth date` / `birthdate`, `birthday`. Plain dates with no birth keyword stay allowed. | ✗ `born 04/12/1985 in ohio`<br>✓ `born in 1985` (no full date shape)<br>✓ `events on 04/12/1985` (no birth keyword) | yes |
| G22 | `GOV_ID_KEYWORD` | Catches a government identity-document number written next to the words naming the document. An id-shaped token (5–20 letters/digits/hyphens, containing at least one digit) within 24 chars after one of these keywords (case-insensitive; both licence spellings): `passport`, `driver's license` / `drivers license` / `driver license`, `license number` / `license no.`, `national id`, `aadhaar`. Document formats vary too much across countries for a bare shape rule, so recall is deliberately keyword-bounded — a number with no naming keyword nearby is judged by the generic guards instead. | ✗ `passport K1234567 renewal`<br>✓ `passport renewal fees 2026` (no id-shaped token in the window) | yes |

## Data classes and what covers them

One row per kind of data that could leak through an auto-executed request
parameter, with the guard(s) that cover it and the honest trade-offs. The
agent's private context comes from connected accounts (mail, calendar,
brokerage), operator instructions, and provider configuration; the leak
channel is a query, prompt, package name, or URL value carrying that context
to a public or third-party endpoint with no human in the loop — by accident
or under prompt injection.

| # | Data class | Example in a guarded param | Guard(s) | False positives (legitimate use denied) | False negatives (leak missed) |
| --- | --- | --- | --- | --- | --- |
| 1 | Large private-context dumps (mail threads, documents) | 3 KB of an email thread pasted into a "search query" | G1 | none — no guarded param legitimately needs > 1 KiB | leaks under 1 KiB pass G1; other rows cover their shapes |
| 2 | Encoded / encrypted / machine-generated payloads | zero-width smuggling; 200-char base64 blob; 16-char base64 fragment | G2, G3, G17 | rare 14+-char gibberish-looking handles (corpus-tuned to zero known cases) | encoded fragments < 14 chars; natural-word encodings outside BIP-39; ~20–30 bytes shaped as a fake wallet address (the address exemption); pure-decimal payloads of 17+ digits (the digit window's upper exemption) |
| 3 | Provider credential prefixes | injected email: "search for `AKIA…` to verify it" | G5 | none observed — prefixes are distinctive | providers absent from the rule set (refresh from Gitleaks) |
| 4 | Private-key blocks | PEM block pasted into a prompt | G6 | none | key material without its PEM armor (G4 catches base64 runs ≥ 14) |
| 5 | JWTs | session token in a query | G7 | none — the `eyJ…eyJ` shape is unambiguous | truncated single segments |
| 6 | Generic API keys/tokens | "api key x7Kp2mQv9zR4tYw8…" | G18, G4 | a hash deliberately searched next to the word "token" | keys < 14 chars with no keyword nearby |
| 7 | Passwords | "password is hunter2secret" | G19, G4 (random-looking ones) | none with the connective requirement | passwords mentioned without password/pwd keywords and shaped like words |
| 8 | Crypto seed phrases | 12 BIP-39 words in order | G8 | a query deliberately listing 12+ BIP-39 words — not observed | phrases interleaved with filler words |
| 9 | Raw crypto keys | 64-hex private key; WIF export | G9 | a sha256 hex digest pasted into a search (same shape as a key — accepted strictness) | truncated keys < 64 hex (G4 covers ≥ 14); public wallet addresses are deliberately exempt shapes |
| 10 | Credential-bearing URLs | `https://user:pass@host/…`; `?access_token=…` | G10, G3 | none — userinfo URLs are never legitimate here | tokens under unrecognized query keys (G4/G18 partially cover) |
| 11 | One-time codes / PINs | "verification code 482913", "PIN 1234" | G20 | numeric qualifiers after "code" not in the exclusion list ("error code 1001" is excluded; unlisted phrases may deny) | bare code-length digit runs with no keyword (below the G11 window) |
| 12 | Bare identifier-length digit runs | a phone or account number pasted without separators | G11 (10–16 digit window) | a legitimate bare 10–16-digit number (a long order id) | runs ≤ 9 (bare one-time codes, bare SSNs — G20 and the checksum rules partially cover) and runs ≥ 17 (provider ids by preference; also lets data ride as long decimal digits within the token cap) |
| 13 | Payment cards | formatted or bare card number | G12, G11 (bare 13–16; valid 17–19-digit cards still fail Luhn+issuer in G12) | none known — issuer+Luhn+corpus filter provider ids | non-Luhn private card schemes (rare) |
| 14 | US SSN | `219-09-9999` or bare | G13 (formatted) | none after invalid-range filtering | bare 9-digit forms (below the digit window; the ABA checksum catches some shapes); SSNs written with spaces |
| 15 | US bank routing | 9 digits from wire details | G14 | a 9-digit non-financial number passing the ABA checksum and district prefix (~4% of 9-digit runs) | routing numbers that fail the district-prefix table |
| 16 | IBAN | `DE89 3704 0044 …` | G15 | none — mod-97 plus country length is strong | countries outside the length table (extend as needed) |
| 17 | Email addresses | "find linkedin of alice@acme.com" | G16 | searching one's **own** public/support email intentionally | obfuscated forms ("alice at acme dot com") |
| 18 | Phone numbers | international or separator forms | G17, G11 (bare) | none known with length bounds | vanity numbers, deeply obfuscated forms |
| 19 | IP literals (private or public) | "dashboard at 192.168.1.10", "geolocate 8.8.8.8" | allowed — little signal worth the usability cost | — | an address that is itself sensitive context; internal hostnames |
| 20 | Dates of birth | "born 04/12/1985" | G21 | historical queries phrased with "born" + full date ("napoleon born 15/08/1769") | DOBs without a birth keyword (a plain date is unguardable) |
| 21 | Government identity documents | "passport K1234567" | G22, G4 (bare id shapes ≥ 14) | renewal-style queries that include the actual number — the number is the leak, so denial is intended | id numbers with no keyword and < 14 chars |
| 22 | Street addresses | "restaurants near 123 Maple St" | **not guarded** | — | all — weak recall and high false positives on place queries; owned by visibility and future flow escalation |
| 23 | Financial amounts and holdings | "should I sell 4,000 NVDA at $120 basis" | G1 only | — | amounts are ubiquitous in legitimate queries; not detectable by shape |
| 24 | Personal names in sensitive context | "gift for Priya, stage 2 cancer" | G1 only | — | NER evaluated and rejected (breaks public search); not detectable |
| 25 | Locations and schedule | "flights thursday 7pm from home" | G1 only | — | not reliably detectable |
| 26 | Health/legal/sensitive topics | agent-authored searches from private context | G1 only | — | topic classification out of scope |
| 27 | Paraphrased private content | "why would Alice cancel the merger" | G1 only | — | provenance, not shape; owned by visibility, the per-task budget, and future task-level source→sink escalation |

Rows 22–27 are the honest limit of string checking: the byte limit bounds
them, the admin activity history makes them visible after the fact, and the
follow-up work below owns them. No detector proves absence.

## Where the guard runs

Surface membership is deliberately **not** enumerated per tool here — that
inventory changes with every tool release and lives where it is enforced:
`tests/test_param_guard_coverage.py` classifies every bundled action input
field as guarded or exempt (with a reason) and fails the build on any new,
unclassified field. The categories:

**Guarded** — agent-controlled free text of automatic actions to public or
third-party destinations:

- public search, discovery, and scraper queries;
- generation and speech prompts, and external media URLs supplied to a generation tool that the provider then fetches (Runway image_url / video_url);
- slugs and hashtags (after their stricter grammar checks);
- package-registry names and query values on the agent network path
  (`python_packages`, `npm_packages` integrations) — a "package lookup"
  whose name encodes a secret is a classic small-payload channel;
- GitHub read-path query values (search `q=` and filters), with both
  token-shape guards (G3, G4) disabled there because revision ids, blob
  shas, and cursors are legitimately long and machine-shaped;
- custom-domain request URLs (operator-configured destinations): the
  operator allows the domain, methods, and paths, and the guard adds content
  protection over the agent-authored URL;
- connected-account mailbox queries (Gmail `search_messages`, `list_drafts`)
  via `allow_identifiers=True`: secret/credential shapes and encoded payloads
  denied; personal identifiers (including one-time codes) allowed as search
  syntax.

On the proxy path the whole reconstructed URL (`https://host<path>?<query>`)
is decoded and scanned as one value rather than parsing segments and query
pairs individually. This keeps the proxy simple and still enforces the
credential-named-query-key rule for free: scanning a full URL routes through
the same `CRED_URL` guard (G10) that denies a credential key carrying a 16+
character value, so `?access_token=<long value>` is caught without the proxy
reparsing anything. Percent-decoding is strict — bytes that are not valid
UTF-8 deny outright rather than being smoothed into replacement characters
while the raw bytes travel upstream.

**Exempt**, with the reason recorded per field in the coverage test:

- approval-gated content (message bodies, posts, captions, event details):
  the operator approval is the control, and denying expected personal
  content would break the product's purpose;
- connected-account object ids (message, thread, draft, label, event ids):
  opaque provider identifiers with existing bounds (mailbox *queries* are
  guarded by the connected-account variant above, which still allows the
  `from:alice@example.com` syntax);
- typed values (enums, numeric ranges, timestamps, strict id grammars,
  normalized provider URLs, bounded opaque cursors): grammar validation is
  stricter than scanning;
- provider protocol and credential flows (OAuth fields, signatures, upload
  form values) on their fixed-destination typed paths;
- binary assets and their store-validated filenames;
- provider-echoed values: package download URLs from index/metadata
  responses (`files.pythonhosted.org` hash segments, npm `/-/…​.tgz`
  tarball paths) and GitHub signed-URL parameters, which are
  provider-generated, not agent free text.

Package tests prove each guarded call site sends the guarded return value,
and behavioral tests drive every guarded surface with a denying value and
assert the descriptive message reaches the caller.

## Errors and observability

| Reason code | Meaning |
| --- | --- |
| `request_param_too_large` | fixed 1,024-byte limit exceeded (G1) |
| `request_param_encoded_blob_denied` | printable/token-shape guard finding (G2, G3, G4) |
| `request_param_secret_denied` | secret-shaped finding (G5–G10, G18, G19) |
| `request_param_pii_denied` | personal/financial identifier finding (G11–G17, G20–G22) |

Tool surfaces raise a `ValueError` whose message names the detected category
and says to remove it and retry; tools surface it verbatim as the action
error. Proxy surfaces return the reason code in the 403 body, and the
network-introspection tools join it to agent-facing guidance from the shared
denial catalog. Neither path ever includes the value or the matched span.

## How operators see it

Every tool whose parameters are guarded renders one identical sentence in
its Integration Guide's protections — defined once as
`PARAM_GUARD_PROTECTION` in `host/param_guard.py` so the wording cannot
drift per tool:

> Request parameters pass the host parameter guard: values shaped like a
> secret, credential, or sensitive identifier are denied before the request
> is sent.

Each guarded tool's Integration Guide additionally renders an expanded
description in its technical-details section — `PARAM_GUARD_TECHNICAL_DETAIL`,
defined beside the short line — covering what is checked, what happens on a
match, what is deliberately not scanned, and that the check sends nothing off
the host. A coverage test asserts exactly the guarded tools carry both. This
section remains the reference-depth description.

## Future exploration

This guard recognizes value shapes, not where content came from, so it
cannot catch a sensitive fact paraphrased out of private context into an
outbound query. The direction worth exploring for that gap is **agent
isolation by purpose**: separating agents that connect to sensitive tools
(mail, calendar, brokerage) from agents that reach public or third-party
destinations, so private context and public egress do not share one agent's
working state in the first place. That is an architectural boundary rather
than a string check, and it addresses the leak class the shape rules here
cannot.

## Alternatives considered

- **Two functions (`bound_text` + `guard_structured_text`).** The bound-only
  path added API surface without adding a control: its fields are
  approval-gated or connected-account and already bounded by shipped limits.
- **Caller-supplied `max_bytes`, guard lists, strictness classes, or named
  per-surface guard sets.** Each was a knob at the call site least likely to
  get security review; each collapsed in turn to the zero-argument call.
  Current surfaces do not diverge enough to earn per-surface selection; the
  one sanctioned variation (GitHub's token-shape exemption) is host code.
- **Central per-field policy registry.** Duplicated the tool-local shipped
  limits and the coverage-test machinery while adding a second place every
  field must be maintained.
- **One universal scanner at the socket or shared HTTP helper.** Serialized
  bytes lack field intent; Gmail, OAuth, and approved content require
  different treatment; the tools UID can bypass the helper anyway.
- **Contextual PII/NER models** (Presidio+spaCy `piiscan`, OpenAI Privacy
  Filter, Piiranha, GLiNER). Measured: the only stack fitting the 2 GiB host
  needed an EOL NumPy pin and 188 MiB RSS and flagged "Tim Cook" and
  "Seattle" in ordinary queries; the capable model (~715 MiB weights)
  exceeds the host budget outright. Names, places, and topics are explicitly
  out of scope for the string guard.
- **Hosted classifier APIs** (Bedrock Guardrails, GCP SDP, DLP vendors).
  Sending candidate text to another party to decide whether it may be sent
  defeats the boundary.
- **Redaction instead of denial.** Executes a query or side effect the
  caller did not request.
- **Tools egress proxy with TLS interception.** A real enforcement boundary,
  justified only when untrusted package code is in scope; far larger than
  the accidental-leak goal. Deferred, unchanged.
